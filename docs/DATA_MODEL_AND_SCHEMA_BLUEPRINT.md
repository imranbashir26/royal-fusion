# Data Model and Schema Blueprint

This document specifies future additive schema work. It does not replace or apply existing migrations.

## 1. Conventions

- Primary keys: UUID generated by PostgreSQL unless an external immutable ID is required.
- Time: `timestamptz` in UTC; display in the configured Pakistan time zone.
- Money: `numeric(12,2)`, never floating point. Currency is `PKR` at launch.
- Names and slugs: trim on write; case-insensitive unique indexes for codes, slugs and SKUs.
- Soft lifecycle: `draft`, `published`, `archived` for products; `active` flags for variants and assignments.
- Mutable rows include `created_at` and `updated_at`; financial records are append-only except controlled status fields.
- Audit records include actor, permission, target, action, safe before/after summaries, request ID, IP hash/metadata and timestamp. Secrets and full payment payloads are excluded.

## 2. Identity and Authorization

### profiles

`id -> auth.users.id`, display name, email/phone mirrors, account status, marketing consent timestamps, internal-note visibility restricted to authorized admins, created/updated timestamps.

### customer_addresses

Customer, label, recipient, phone, address lines, city, province, postal code, country `PK`, default flag and timestamps. One default address per customer is enforced with a partial unique index and transactional replacement.

### roles, permissions, role_permissions, user_roles

Keep the existing relational permission model. Permission checks use immutable permission keys, never role-name comparisons. User-role assignments include active state, assignment/revocation actor and timestamps.

### auth_sessions metadata

Optional application session registry for administrators: Auth session identifier hash, user, created/last-seen/revoked/expiry timestamps, user-agent summary and revocation actor. Do not store access tokens.

## 3. Catalog

### products

Required fields: ID, name, slug, short/full description, fragrance family, top/middle/base notes, concentration, longevity, sillage, occasions, inspired-by text, status, published timestamp, featured flag, manual bestseller pin/rank, SEO title/description, created/updated timestamps.

Rules: products with order-item references cannot be hard-deleted. `published` requires a name, slug, active variant and primary media asset. Bestseller sales totals are derived from eligible delivered/completed order items; manual fields only influence presentation.

### variant_attribute_definitions

ID, key, label, display order and active state. Initial rows are Size or volume, Type and Packaging, but none are hardcoded in frontend logic.

### variant_attribute_values

Definition ID, normalized value, display label, display order and active state. Unique `(definition_id, lower(value))`.

### product_variants

Product ID, stable ID, SKU, regular price, sale price, cost price, stock quantity, low-stock threshold, weight in grams, active state, display order and timestamps.

Constraints: regular price > 0; sale price is null or > 0 and <= regular price; cost price >= 0; stock and threshold >= 0; weight > 0 when provided. SKU is case-insensitive unique. Ordered variants are deactivated, not deleted.

### product_variant_values

Variant ID and attribute value ID with a composite primary key. A generated/canonical attribute signature is unique within each product to prevent duplicate combinations.

### categories, collections

Name, slug, description, status/active state, display order, SEO metadata and media references. Collections additionally include featured flag, visibility start/end and banner references. Initial category seed: Men, Women, Unisex, Attars, Gift Sets and Testers.

### product_categories, product_collections, related_products

Join tables use composite unique keys and display ordering where relevant. Related products prevent self-reference and normalize each directed relationship once.

### media_assets and media_usages

Cloudinary public ID, resource/version metadata, secure URL, folder, media type, width, height, format, bytes, alt text, focal point X/Y, status and timestamps. Usages relate an asset to a product, variant, category, collection, hero slide, campaign, review or testimonial with role and display order. A referenced asset cannot be deleted.

## 4. Homepage and Promotions

### homepage_sections

Stable section key, title, visibility, display order, start/end, configuration JSON validated against a server-owned schema and timestamps. Allowed keys are announcement, hero, featured_products, best_sellers, new_arrivals, featured_collections, promotional_banners and testimonials. Arbitrary HTML, JavaScript and CSS are prohibited.

### announcement_bars

Text, optional CTA label/destination, active flag, start/end and display priority. Enforce title/CTA limits from the admin specification.

### hero_slides

Desktop/mobile media IDs, title, description, CTA label/destination, alignment enum, display order, active state, start/end, `autoplay_interval_seconds` and timestamps. Each slide interval is constrained to 4-10 seconds and defaults to 6 seconds.

### promotions and coupons

Promotions schedule product/collection/site presentation. Coupons support percentage, fixed cart, fixed eligible-product and free shipping behavior; minimum subtotal, maximum discount, start/end, total/per-customer limits and active status.

Join tables scope coupons to products/categories. Redemptions reference customer when authenticated and normalized guest email hash where policy allows. A unique reservation/redemption key prevents duplicate use by one order.

## 5. Inventory

### inventory_movements

Variant, signed quantity delta, resulting balance, reason enum, reference type/ID, note, actor and timestamp. Reasons: initial, admin_adjustment, order_deduction, cancellation_return, refund_return, damage, correction and import.

The variant stock balance changes only in the same transaction that writes a movement. Manual changes require reason and permission. Inventory movements are never edited or deleted.

## 6. Orders and Fulfilment

### orders

Order ID/number, optional customer ID, immutable contact and shipping snapshots, order/payment/shipping statuses, currency, tax-inclusive subtotal, discount, shipping, tax-display amount, total, coupon/shipping method, payment method, idempotency key, cancellation request fields, cancellation approval fields and timestamps.

Order number format is `RF-YYYY-000001`, generated by a yearly transactional sequence. The internal UUID remains the primary key.

### order_items

Order, product/variant references, immutable product/variant/SKU/attribute snapshots, quantity, unit regular/sale price, discount allocation, tax-display allocation and line total. Quantity > 0 and monetary amounts >= 0.

### order_status_history and order_notes

Append-only transitions with from/to status, actor, reason and timestamp. Notes have visibility (`internal` or `customer`) and author. Customer-visible notes must be separately permissioned.

### payments and refunds

Payments contain method, provider reference, amount, status and safe metadata. Bank-transfer verification records verifier/time. Refunds reference payment/order, amount, reason, actor and timestamps. Total successful refunds cannot exceed captured/paid amount. Card-provider payloads are never stored unredacted.

### shipments and invoices

Shipments store courier, tracking reference, shipping state and delivery timestamps. Invoices store immutable number, legal-field snapshot, totals and generated document reference. Updating store settings never alters past invoices.

### cancellation_requests

Order, requester, reason, status, reviewed actor/time and resolution. Only pre-shipment orders can be approved for cancellation. Approval and stock restoration occur transactionally.

## 7. Reviews, Testimonials and Configuration

### reviews

Product, optional variant/customer/order item, rating 1-5, title/body, moderation status, verified-purchase flag derived from order ownership, admin response, response actor/time and timestamps.

### testimonials

Attribution, city/label, quote, rating, optional media, visibility, display order and moderation timestamps. Avoid publishing private customer identifiers.

### public_site_settings

Only browser-safe values: store display identity, support contacts intended for publication, currency, locale, delivery estimate, enabled public payment labels and other allowlisted presentation settings.

### private_site_settings

Owner-only operational configuration references and protected values. Provider credentials remain deployment secrets, not table fields. Tax settings, legal invoice values and sensitive payment configuration are never publicly selectable.

### shipping_methods and shipping_rates

Default Pakistan method with PKR 300 charge, free-shipping threshold, estimate and active state. Optional city/zone rates override the default and include COD availability. Overlapping active rules of equal priority are rejected.

### email_outbox

Event type, aggregate ID, recipient reference, template key, payload containing only required render data, status, attempts, next attempt, provider message ID and timestamps. Checkout commits the event; a dispatcher sends it later. No provider secret is stored.

## 8. Status Enums

- Product: `draft`, `published`, `archived`.
- Order: `Pending`, `Confirmed`, `Processing`, `Shipped`, `Delivered`, `Cancelled`, `Refunded`.
- Payment: `Unpaid`, `Pending Verification`, `Paid`, `Failed`, `Refunded`, `Partially Refunded`.
- Shipping: `Not Fulfilled`, `Preparing`, `Dispatched`, `In Transit`, `Delivered`, `Returned`.
- Cancellation request: `Requested`, `Approved`, `Rejected`, `Withdrawn`.
- Review: `Pending`, `Approved`, `Rejected`.

Exact transition rules are defined in `ORDER_CHECKOUT_AND_INVENTORY_LIFECYCLE.md` and must be enforced by transactional backend functions, not UI controls alone.

## 9. Index Recommendations

- Case-insensitive unique: product/category/collection slugs, SKU, coupon code and role/permission keys.
- Catalog: product `(status, published_at desc)`, variant `(product_id, active, display_order)`, joins by both foreign-key directions.
- Search: PostgreSQL full-text or trigram indexes for product name/slug/SKU; never `%term%` scans over serialized rows.
- Orders: unique order number/idempotency key; `(customer_id, created_at desc)`, `(order_status, created_at desc)`, payment/shipping status and normalized email/phone lookup.
- Inventory: `(variant_id, created_at desc)` and `(reason, created_at desc)`.
- Campaigns: `(active, start_at, end_at, display_order)`.
- Reviews: `(product_id, status, created_at desc)`.
- Audit/outbox: actor/target/timestamp and `(status, next_attempt_at)`.

## 10. RLS and Ownership Rules

- Anonymous users read only published catalog, active homepage data, approved reviews and allowlisted public settings.
- Anonymous and authenticated browsers cannot insert orders, payments, refunds, inventory, media or audit records directly.
- Customers read/update their own profile and addresses and read only their own linked orders, items, notes, shipments, invoices and cancellation requests.
- Administrators require an active user-role assignment and the exact permission key. Cost prices, private settings, refunds and customer private fields use separate permissions.
- Service-role operations run only in Express. The service credential bypass does not replace backend authorization.
- Blog data is outside Supabase after Sanity integration.

## 11. Proposed Additive Migration Sequence

1. Normalize dynamic attributes/variant values and add cost, threshold and weight fields.
2. Add media asset registry/usages and variant-specific media.
3. Add homepage sections, announcement bars, hero slides and campaign constraints.
4. Extend customers, reviews and testimonials.
5. Extend fulfilment, cancellations, invoices and status constraints.
6. Add email outbox and delivery indexes.
7. Add search/reporting indexes and constrained aggregate views.
8. Add/adjust RLS, grants, RPCs and verification tests for every new object.

Each migration requires a rollback procedure, staging rehearsal, data backfill checks and a post-deployment read-only verification script. Existing migration history must not be rewritten.
